## 三、技术架构分析

### API 架构

```
API Route (route.ts)
  └─ withAuth() 包装器
      ├─ 自动 catch + 统一错误响应
      ├─ 认证检查 (getCurrentUser)
      │   ├─ X-Dev-User-Id header → 开发模式
      │   ├─ dev_user_id cookie → 开发模式备用
      │   └─ Supabase session → 生产模式
      └─ 传入 { request, user, params }
```

**API 工具链**:
- `createApiResponse()` / `createApiError()` / `createUnauthorizedResponse()`
- `createPaginatedResponse()` + `getPaginationParams()` (已修复 NaN/boundary bug)

### 路由保护架构

```
请求 → middleware.ts
  ├─ 公开路径 (/login, /api/auth) → 放行
  ├─ API 路由 → 放行 (由 withAuth 处理)
  ├─ 受保护路径 (home, ai, capture, ...) → 检查 dev_user_id cookie
  │   ├─ 无 cookie → 重定向 /login
  │   └─ 有 cookie → 放行
  └─ 静态资源 → 放行

页面渲染 → ProtectedRoute
  ├─ useUser() → 检查用户登录状态
  │   ├─ 加载中 → LoadingSpinner
  │   ├─ 已登录 → ErrorBoundary → 页面内容
  │   └─ 未登录 → redirect(/login)
  └─ ErrorBoundary (类组件)
      ├─ 无错误 → 渲染子组件
      └─ 有错误 → 错误 UI + 重新加载按钮
```

### 认证模式

项目目前**完全运行在开发模式下**：
1. 登录页面输入手机号 → 发送验证码 → 验证后存入 localStorage
2. `syncDevAuthCookie()` 将用户 ID 同步到 cookie
3. `middleware.ts` 检查 cookie 保护路由
4. API 层 `getCurrentUser()` 优先检查 header → cookie → Supabase session

### AI 服务架构

```
ai-services.ts
├─ callDeepSeek()       ─ 文本生成 (备用)
├─ callDoubaoChat()     ─ 多模态理解 (主力)
├─ callDoubaoVision()   ─ 图片分析
├─ generateImage()      ─ Seedance 图片生成
├─ submitVideoTask()    ─ Seedance 视频生成
├─ getVideoTaskStatus() ─ 视频生成状态轮询
├─ summarizeContent()   ─ AI 总结灵感
├─ generateCopywriting()─ AI 文案生成
└─ logAiUsage()         ─ AI 用量记录
```

### 需要关注的问题 ⚠️

| 问题 | 位置 | 影响 |
|------|------|------|
| **测试覆盖率极低** | 全项目 | 仅 2 个工具模块有测试 |

| **Axios 依赖未使用** | `package.json` | 依赖 axios 但 API 客户端使用 fetch |
| **几处未使用的导入** | home(Flame), hotspot(Settings), capture(重复图标) | 代码整洁度 |
